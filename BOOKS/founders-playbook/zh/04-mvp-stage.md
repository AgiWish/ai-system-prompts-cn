# MVP 阶段

---

许多创始人把 MVP 阶段当作构建阶段，但它本质上仍然是一个收集证据的阶段。区别在于，你现在收集的是关于解决方案的证据，而不是关于问题空间的证据：是否有一群真实、可识别的人觉得这个产品足够有价值，愿意使用、回来继续使用、付费，或者向他人推荐。

## MVP 阶段目标

作为 AI 原生创业公司的创始人，你的目标是把已经验证的问题转化为一个真实用户会实际使用的工作产品。它不是带有路线图上所有功能的完整版本，而是你想法中最小、最聚焦的版本：把真实解决方案放到真实用户面前，并产生关于产品-市场匹配的真实证据。

同时，你现在如何构建，会决定以后能做什么。因此，MVP 阶段还有第二个同样重要的目标：在保持速度的同时，不积累那种会复利增长、并在真实用户大规模到来时反噬你的技术债。

最后，从第一天起就投资持久上下文，是让 AI 成为倍增器而不是熵源的关键。在 AI 原生创业公司中，代码库是你一次又一次和 AI 协作的对象，因此可读性是基础。跳过规格说明、架构决策和 CLAUDE.md 这类上下文文件的创始人，会撞上一堵可预期的墙：每次新会话都要重新解释代码库，AI 生成的改动也会逐渐偏离最初愿景。

## MVP 阶段退出标准

MVP 阶段的退出条件是真正的产品-市场匹配证据：证明一群具体、可识别的用户认为产品足够有价值，愿意回来使用（留存）、付费（收入），或告诉其他人（推荐）。

## MVP 阶段挑战

MVP 阶段，创始人的首要原则是速度和判断。挑战集中在：你能否足够快地以正确方式构建正确的东西，同时不偷那些以后会付出代价的懒。

## Agentic 技术债

挑战在于：AI 基本移除了过去控制什么能进入生产环境的自然瓶颈，速度几乎是确定的。但如果创始人在 MVP 构建中只考虑速度，就会积累自己难以偿还的技术债。

某些技术债在 MVP 阶段是合理的，前提是你理解它必须在规模化前被管理。传统技术债会逐步积累，可以随着时间或专门 sprint 被清掉。但 AI 技术债会复利增长。

如果没有写在某个 AI 可读位置的规格和架构约束，每次会话都会从零重新推导基础决策，而这些决策会漂移。最终你会得到一个没有一致心智模型的代码库。问题不一定是某个单独部分很差，而是这些部分从未被设计成相互配合。这是一个真实问题，而且常常很晚才暴露。

## 误判产品-市场匹配

挑战在于：AI 工具可以生成漂亮的早期数字，但这些数字并不保证市场需要你的产品。

早期势能对创始人有很强心理影响。经过数周或数月验证和纪律化构建后，产品上线会让人感觉“我一直是对的”。Agentic coding 工具可以让你比以往更快到达这一刻，但早期 traction 并不等于产品-市场匹配。

发布热度可能来自短暂因素：创始人的朋友、投资人其他 portfolio 公司的潜在买家，或 Hacker News 首页带来的流量峰值。这些都不能可靠预测第 6 周或第 12 周，当最初推动力消退后会发生什么。

## 零摩擦范围蔓延

挑战在于：当构建感觉毫不费力且几乎免费，总会出现“再加一个很酷的功能”或“再处理一个边界情况”的诱惑。这种 scope creep 可能弊大于利。

范围蔓延一直是创业风险。不同之处在于，过去约束它的工程时间成本已经不再以同样方式存在。现在加一个功能可能只需要一个下午，而不是一个 sprint。

每个新增项在当下都显得合理：当然产品应该处理这个边界情况，当然用户会想要这个流程。它们之所以不像范围蔓延，是因为每一项用 Agentic coding 构建都太容易。但当产品扩展到原始边界之外，你会失去方向和动能。

解法是在构建前写下范围定义：产品做什么、刻意不做什么，以及什么来自真实用户的证据足以证明应该增加新内容。这会把决策点从“我们要不要做这个？”移动到“是否有足够多用户明确说，没有这个他们无法获得价值？”

## 缺乏经验导致的不安全

挑战在于：创始人使用 AI 工具快速把应用推向市场，但如果没有先理解基本安全原则，就可能让用户暴露在本可避免的风险中。

残酷事实是，Agentic coding 工具会生成能运行的代码，不会天然生成安全代码。功能代码很容易判断，因为它要么能用，要么不能用。安全漏洞则通常不可见，直到被利用。首次创业者没有自然反馈回路提醒自己哪里不对。但把 MVP 发布给真实用户，就意味着真实数据、真实暴露面和真实后果。

忽视安全不是 AI 原生项目才有的问题。各时代的自举创业公司都常把安全考虑推迟到构建后期，有时甚至等到生产发布前夕。任何用户触碰产品前进行安全审查，是把 MVP 负责任地发布到世界上的最低门槛。

## Claude 如何帮助 MVP 阶段创始人

## 构建前定义架构

在 Claude Code 写第一行生产代码之前，先用 Claude 定义并记录会约束整个 MVP 构建的架构决策：遵循哪些模式、避免哪些依赖、接受哪些权衡以及为什么。这份输出会成为聚焦的架构上下文文档，也是 Claude Code 运行时的护栏。

没有这份上下文，每次会话都会从零开始，Claude Code 只能自行推断结构假设。让 Claude Code 在没有护栏的情况下构建，可能得到一个功能上可用但结构上不一致的代码库，而迭代和规模化这种代码库最终是在浪费时间和 token。迟早会到达一个代码结构坍塌、不得不重建的点。

• 练习：打开 Claude，描述你正在构建什么：它解决的核心问题、服务的用户、未来 6 个月现实预期的规模。让它帮助你定义 MVP 构建应遵守的架构原则、在约束下应避免的依赖，以及你有意识接受的权衡。

接着把输出保存为 CLAUDE.md。这是你的架构上下文文档，是构建中的第一份 artifact，也是后续每次会话依赖的文件。CLAUDE.md 为 Claude Code 提供项目级说明，是项目的持久记忆。

## 定义并执行 MVP 范围

AI 时代 MVP 的典型失败模式之一，是没有摩擦的范围蔓延。正如你要定义并记录应用架构，也需要在构建任何功能前定义 MVP 范围。

Claude 可以帮助创建范围文档，说明产品做什么、刻意不做什么，以及功能变更标准：在这个阶段，什么来自真实用户的具体证据才足以 justify 新增功能。

当新功能想法出现时，使用 Claude 压力测试：它是真实用户信号，还是创始人热情披着产品思考的外衣。

## 用 Claude Code 构建 MVP

一旦架构和范围定义完成，Claude Code 就成为主要 MVP 构建工具。用它生成、测试、调试和迭代代码库，但要把每次会话视为执行已经做出的产品决策，而不是随手加入新点子的机会。

每次 Claude Code 会话开始时，先回顾范围文档，并提供 CLAUDE.md 架构上下文。结束时，把会话中出现的新决策更新进去。目标是拥有一个你能解释其结构的代码库，而不只是一个能运行的代码库。

• 练习：为 Claude Code 工作创建简单会话模板，包括架构上下文文档、本次具体任务，以及需要遵守的约束和模式。每次会话结束后，在上下文文档中添加简短日志：构建了什么、做了什么决策、引入了什么假设。

## 任何用户触碰前先做安全审查

作为 AI 原生创业创始人，你有责任知道代码库里有什么，理解潜在暴露向量，并避免把明显漏洞交给信任你的真实用户。

Claude 可以对 AI 生成代码做有用的第一轮安全审查，识别常见漏洞。这应该成为发布前循环中的习惯。但它不是安全工具或专业人工审查的替代品；风险更高时尤其如此。

Claude Code Security 进一步扫描代码库安全漏洞，并为人工审查提供定向修复建议。注意，在本电子书发布时，Claude Code Security 仍是有限 beta，因此在把它纳入工作流前要检查当前可用性。

• 练习：在部署给任何真实用户前，用明确 brief 让 Claude 审查核心应用代码：认证和会话处理、API 响应中的数据暴露、输入校验和注入风险，以及有已知漏洞的依赖。认真对待每个发现；凡涉及认证、密钥或数据处理的问题，都应有人类复核。

## 发布前建立度量框架

误把早期 traction 当作 PMF 的创始人，通常也是发布后才开始追踪数据的人，而且选择的指标常常是为了证明什么有效，而不是揭示什么无效。解法是在第一个用户出现前建立度量框架。

用 Claude 定义对你的具体产品最重要的指标、基准是什么，以及哪些数据模式能构成真实 PMF，而哪些只是好看的噪声。尤其要在发布 MVP 前设定留存基准、激活标准，以及第 7 天和第 30 天目标。

然后定义你的产品中“假阳性”长什么样：例如有注册但无激活、有收入但无留存、有初始热情但无重复使用。当数据到来时，让 Claude 从怀疑者角度反驳你的 traction：这些数字可能说明不了什么？

## 管理用户发现与反馈物流

一旦真实用户进入产品，运营层会快速扩大。Claude Cowork 可以处理重要但繁琐的工作：建立和维护用户联系人列表、运行外联序列、安排反馈会议、分流 bug 报告、跟踪迭代周期。想法阶段管理发现物流的 MCP 集成，在这里同样适用。

但对于细腻的用户反馈解释，人类必须留在循环中。例如用户说“这个很好，但我希望它还能……”，需要判断：这是核心需求还是锦上添花？是某个客户特例还是某个细分群体代表？缺失功能是真问题，还是 onboarding 上游出了问题？没有工具能自动回答这些问题。

• 练习：配置 Claude Cowork 运行 MVP 阶段反馈循环：向早期用户列表起草外联，安排反馈会议，设计 bug 和功能请求的结构化 intake 流程，并每周汇总收到的反馈。你先自己审阅汇总，再让 Claude 分析是否漏掉重要点。

## 朝证据迭代，而不是朝完整性迭代

MVP 阶段结束于真实 PMF 证据，而不是产品感觉多么“完成”。宣布达到 PMF 并进入发布阶段，本质上是结合创始人直觉和收集证据的判断。但有几个有用测试：

• Sean Ellis 测试：询问活跃用户：“如果你无法再使用这个产品，你会有什么感受？”如果超过 40% 回答“非常失望”，这是一个有意义的 PMF 指标。

• 努力测试：PMF 前，留存需要持续干预，包括频繁外联、激励、个人跟进，以及创始人大量精力。PMF 后，产品开始自己拉动用户。当事情从“推”转为“拉”，这是非常清晰的信号。

没有单一数据点能确认 PMF。它必须在多个迭代周期中持续成立。

## 证据要求时就转向

如果投入了这些工作后仍然无法接近 PMF，当前结果没有确认你最初方向并不是失败，而是系统在工作：MVP 阶段本来就是为了在你过度投入错误答案之前暴露信息。

当数据不支持当前产品时，用 Claude 分析数据在告诉你什么：也许没有转化的用户根本不是正确目标；也许正确受众已经在数据中，只是被低估；也许受众正确，但 MVP 的价值表达没有打动用户，需要调整 onboarding、信息传达或核心功能强调。

也要保持开放：这种断裂可能深到需要更根本的改变。

• 练习：如果你完成了三个或更多迭代周期，仍然没有朝 PMF 基准取得有意义进展，把留存数据、用户反馈和原始问题假设交给 Claude，问三个问题：数据中是否有某个细分群体反应不同？设计价值和体验价值之间的差距是定位问题还是产品问题？当前产品要找到真实 PMF，必须有哪些前提成立，而这些前提在当前数据下是否现实？

让答案决定你是调整、转向，还是回到想法阶段。
