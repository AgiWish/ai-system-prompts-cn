# 发布阶段

> 翻译方式：章节级人工初译整理

如果 MVP 阶段是在证明你的产品值得存在，那么发布阶段就是证明你的业务值得增长。

## 发布阶段目标

在发布阶段，创业者必须把早期 traction 转化为可重复、可持续的增长引擎。除了让产品达到生产可用，你还必须加固底层基础设施，并围绕产品构建一家真正的公司。

想法和 MVP 阶段的创业公司天然以创始人为中心，因为你需要完整的场景感知和紧密反馈循环。但到了发布阶段，如果创始人仍然试图亲自抓住每一条线索，就会成为瓶颈。目标不是把自己从公司中移除，而是建立运营系统，把你的注意力释放给只有创始人能做的决策。

## 发布阶段退出标准

发布阶段的退出条件有三部分：

1. 增长是可重复且由渠道驱动的。你不仅能留住用户，还能通过具体渠道可预测地获得用户，并理解单位经济：CAC、LTV 和回本周期都是你知道且能解释的数字。

2. 产品能够承载生产工作负载。基础设施已经加固，安全与合规有序，可靠性能在真实生产条件下成立，而不只是你测试过的条件。

3. 运营不再被创始人卡住。流程已经存在，自动化已经到位。你不再亲自处理支持、分流、sprint 规划或报告。

## 发布阶段挑战

找到 PMF 是早期创业生命周期中最难的问题。现在，创始人的挑战变成了保持它。发布阶段中，即使公司找到了真实产品 traction，也可能因为围绕产品的组织无法跟上而崩塌。

## 技术债到期

挑战在于：为了速度和验证而构建的 MVP 代码库，曾经足以证明产品可行；但生产流量、新功能和复杂性增长，会开始暴露当初的捷径。

在 MVP 阶段，积累一些技术债是合理的速度权衡。到了发布阶段，债务开始产生利息；拖得越久，修复越昂贵。

解法包括系统性架构审计，找出结构弱点；定向重构最严重的问题；并显著扩展测试覆盖，防止下一轮功能开发重新引入同类问题。

## 创始人成为瓶颈

挑战在于：MVP 阶段创始人在每个循环中是资产；发布阶段，当支持量增长、产品决策堆积、运营复杂性增加，同样的本能会变成约束。

从亲自做事，转向设计能做事的系统，是创业生命周期中最难的转变之一。因为很少有清晰时刻提醒你“现在该变了”，风险是你完全错过这个转换，继续停留在 builder mode，而组织在你周围停滞。

信号包括：本该一小时完成的决策，要等你一周才处理；支持请求堆积，因为只有你知道答案；运营任务只有在你亲自想起时才发生。

补救方法是全面审计你正在亲自处理的一切，从最小任务到最高风险决策，识别什么可以系统化、什么可以委派、什么确实仍然需要创始人的判断。

## 安全与合规不能再推迟

挑战在于：MVP 阶段保持安全和合规措施简单可以接受；但当真实用户、真实数据和潜在企业合同出现时，这会变成负债。

MVP 阶段，少量 beta 用户和没有敏感生产数据时，安全漏洞还只是理论风险。一旦产品进入生产、有真实用户依赖，风险就变成真实暴露。处理客户数据、支付或进入监管行业时，合规要求也会真正适用。

补救方式是在生产规模到来前，而不是之后，进行系统性安全与合规审查。所有发现都应被视为下一波用户到来前必须处理的修复项，而不是建议。

## 未准备好就扩张

挑战在于：新市场和融资机会看起来像增长机会，但也可能是 PMF 消失的地方。

你已经建立的初始 traction 是真实的，但它也只适用于早期受众。过早进入一个与原市场显著不同的市场，会引入新的用户行为、合规要求、支付基础设施和基本预期，而产品并不是围绕它们设计的。变量突然太多，你会失去清晰解释数据的能力，也可能忽视原有用户群。

## Claude 如何帮助发布阶段创始人

发布阶段会同时用到 Claude 的三种形态，并且它们会彼此支持：一个工具的输出会成为另外两个工具的输入。这种结果会自然复利。创始人把三者结合使用，获得的效果会大于各部分之和。

这也是极精简创业模型在结构上可行的原因：Claude Code 构建产品，Claude Cowork 构建围绕产品的公司，Claude 帮助运营化产品与组织知识，小团队因此可以像更大的公司一样运转。

## 在技术债复利前修复它

MVP 代码库能工作，但需要系统性修复，找出可能变成结构性负债的技术债。

先用 Claude Code 做完整架构审计：识别代码库脆弱处、会变得昂贵的捷径，以及测试覆盖薄弱到下一轮功能会重新引入同类问题的地方。

再把 Claude Code 的审计结果交给 Claude，让它排序和拆解修复工作：哪些必须在下次发布前修，哪些可以等一个 sprint，哪些在当前阶段可以作为可接受债务继续存在。这也是记录 MVP 阶段架构决策的时刻。把它们写进 CLAUDE.md，确保未来每次 Claude Code 会话都从同一理解出发。

• 练习：让 Claude Code 审计 MVP 代码库，输出结构弱点、测试覆盖缺口和重构候选的优先列表。再把清单交给 Claude，让它按多个 sprint 排序：必须先处理的问题、可以和功能开发并行的问题、可以等待的问题。

## 构建替代创始人注意力的系统

要构建释放创始人注意力的运营系统，首先要知道你的注意力正在花在哪里。用 Claude Cowork 对当前运营负载做结构化审计，记录每个重复任务、每个落到你桌上的决策、每个因为你亲自想起才发生的流程。然后把清单分类：哪些可以完全自动化，哪些需要人但不一定是你，哪些确实需要创始人判断。

审计完成后，让 Claude Cowork 设计自动化候选的工作流逻辑：触发条件、决策规则、输出形态，以及完成后流向哪里。

## 把安全与合规变成产品工作流

用 Claude Code 识别 SOC 2、GDPR、HIPAA 或目标市场标准常见的代码级问题。这会暴露漏洞和合规缺口。把发现交给 Claude，帮助你排序修复工作，并设计控制、审计日志和访问管理等企业买家签约前会询问的内容。注意，AI 扫描只是辅助，不能替代合格的合规审查。

接下来，把合规工作流嵌入开发周期，而不是当作一次性项目。合规文档需要持续维护和更新。对接近企业合同或国际市场的创始人来说，这也是使用 Claude Code 安全扫描准备独立安全评估的时刻。

• 练习：让 Claude Code 按目标市场要求做代码级安全审查。把输出交给 Claude，产出两件事：安全修复优先序列，以及通过潜在企业买家合规审查所需的文档和控制清单。

## 建立过去一直跳过的产品管理流程

发布阶段需要一套轻量、可重复的流程，不依赖创始人触发也能运转。用 Claude 设计产品时间线和工作周期如何组织、功能进入 Claude Code 前必须包含哪些规格说明、bug 如何分流、周度指标报告覆盖什么并如何分发。

流程设计完成后，用 Claude Cowork 构建和运行运营层：安排 sprint 仪式、路由 bug 报告、从连接的数据源编译周报、维护用户信号进入产品决策的反馈回路。

• 练习：让 Claude 设计一个轻量产品管理操作系统：sprint 节奏、最低规格模板、bug 分流决策树、从真实数据源拉取的周度指标简报。然后设置 Claude Cowork 来运行这些周期性运营元素。
