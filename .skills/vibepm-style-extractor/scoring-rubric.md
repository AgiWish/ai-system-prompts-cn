# 风格提示词自动评分规则

用于评估“风格提取结果 + 提示词包”的质量。
满分 100，建议 80 分以上再投入批量生成。

## 评分维度（总分 100）

### 1) 风格一致性（20 分）
- 0-8：风格关键词和 token 冲突，整体不统一
- 9-14：基本统一，但局部存在风格漂移
- 15-20：方向明确，组件与布局语言高度一致

### 2) Token 可执行性（15 分）
- 0-6：大量抽象描述，缺少可落地约束
- 7-11：大部分可用，仍有少量模糊项
- 12-15：可直接用于实现，命名和层级清晰

### 3) 色彩与对比（15 分）
- 0-6：对比不足或过度刺眼，语义色不清楚
- 7-11：可读性可接受，层次尚可
- 12-15：对比关系稳健，强调与中性色平衡良好

### 4) 排版与层级（10 分）
- 0-4：字号/字重层级混乱，阅读节奏差
- 5-7：层级基本清晰，仍可优化
- 8-10：层级明确，阅读节奏自然

### 5) 组件完整度（10 分）
- 0-4：组件状态缺失，规则不成体系
- 5-7：有核心组件，但状态覆盖不完整
- 8-10：主流组件与状态规则完整，风格统一

### 6) 布局与间距系统（10 分）
- 0-4：布局描述笼统，无节奏约束
- 5-7：有栅格和间距，但执行边界不清
- 8-10：栅格、容器、间距节奏可复现

### 7) 约束与防跑偏（10 分）
- 0-4：几乎没有 avoid 规则
- 5-7：有部分禁用项，但不足以约束生成
- 8-10：avoid 清晰且针对性强，可有效防漂移

### 8) 变体控制能力（10 分）
- 0-4：变体与原风格差异失控或过小
- 5-7：变体可用，但控制粒度一般
- 8-10：在保留 DNA 的前提下可控变化

## 风险扣分项（可叠加，最低扣到 0）

- 出现自相矛盾约束（如“高对比”同时“弱对比”）: -5
- Token 命名不一致（多套命名混杂）: -3
- 缺失按钮状态（hover/focus/active）: -5
- 只有形容词，没有实现性参数: -8
- 未给出任何防跑偏规则: -6

## 输出模板（评分结果）

写入 `*-style.md` 时须使用下列**中文结构**（顶层标题保留中英并列，与 SKILL 固定格式一致）：

```markdown
## 质量评分 / Quality Score
- **总分**：XX/100
- **结论**：通过 | 需修订 | 不通过（可括号内简短说明，如「可直接用于迭代」）

## 分项得分 / Dimension Scores
- 风格一致性：X/20
- Token 可执行性：X/15
- 色彩与对比：X/15
- 排版层级：X/10
- 组件完整度：X/10
- 布局与间距：X/10
- 防跑偏约束：X/10
- 变体控制力：X/10

## 扣分说明 / Deductions
- ...

## 关键风险 / Key Risks
- ...

## 优先修订（三项）
1. ...
2. ...
3. ...
```

（可选）内部备注：`Pass` / `Revise` / `Reject` 与 **通过** / **需修订** / **不通过** 一一对应。

## 判定标准

- **通过**：80–100（可直接用于生产迭代）
- **需修订**：60–79（建议先修正再批量生成）
- **不通过**：0–59（需重做提取或重写提示词）

## 复评规则

当结论为 **需修订** 或 **不通过** 时：
1. 优先处理 `## 优先修订（三项）` 所列条目。
2. 仅允许一次修改 1–2 个变量（如色彩对比或间距密度），避免同时大改。
3. 修订后必须重新评分，并与上一版对比增量。
