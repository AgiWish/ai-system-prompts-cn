# 组件规范（WEB 官网 / 落地页）

本规范用于约束企业官网、产品介绍页与营销落地页中常用组件的视觉与交互一致性。页面开发时，除业务定制部分外，应优先复用本规范，确保整体风格统一、信息层级清晰、转化路径明确。

## 与视觉规范的协同关系

- 当本规范与 `./frontend-design.md` 同时使用时，本规范优先负责信息架构、组件结构、版面对齐、响应式、可读性与转化路径等硬性约束。
- `./frontend-design.md` 主要用于指导视觉风格、气质塑造、装饰细节、动效表达与差异化呈现。
- 若两者在字体选择、装饰密度、布局自由度、动效强度等方面发生冲突，应优先满足本规范，再在不破坏页面稳定性与业务目标的前提下吸收视觉表达。

## 适用范围

- 企业官网首页：品牌展示、核心能力、客户背书、页脚信息。
- 产品/服务介绍页：功能亮点、方案说明、优势对比、FAQ。
- 营销落地页：活动专题、线索收集、CTA 转化、信任证明。
- 招聘 / 关于我们页面：公司介绍、团队文化、岗位信息、联系方式。

## 通用原则

- 组件服务于信息传达与转化，不做脱离业务语义的装饰堆砌；允许使用适度的装饰性图标、分隔元素与背景细节增强识别度与版面节奏。
- 优先使用 WEB 页面语义，不使用 APP、系统壳、设备框等移动端表达。
- 同一页面中组件风格应统一：圆角、描边、阴影、字体、按钮状态不得混乱。
- 所有颜色、间距、圆角、阴影优先使用全局变量，避免在组件中散落硬编码。
- 组件必须支持桌面端布局，并兼容常规平板与移动端响应式缩放。
- 组件状态需完整，至少覆盖 `default`、`hover`、`active`、`disabled`；表单组件还需覆盖 `focus`、`error`。
- 组件间距应服从区块节奏，不因单个组件放大造成页面信息密度失衡。

## 信息层级与排版

- `H1` 用于页面首屏主标题，通常每页仅出现一次，建议 `40px` 到 `56px`。
- `H2` 用于一级业务区块标题，建议 `28px` 到 `36px`。
- `H3` 用于卡片、能力点、子模块标题，建议 `20px` 到 `24px`。
- 正文主字号建议使用 `14px` 或 `16px`，辅助信息使用 `12px`。
- 同一页面正文档位不超过两档，避免阅读节奏破碎。
- 标题、副标题、说明文字之间需形成稳定层级，避免全部加粗或字号接近。
- 长段落文本控制可读宽度，避免一行过长影响阅读体验。

## 布局容器（Section / Container）

- 每个业务区块使用语义化 `section`，并配合统一容器类控制内容宽度。
- 桌面端内容主容器建议最大宽度为 `1200px` 左右，根据品牌调性可在 `1080px` 到 `1280px` 之间调整。
- 区块上下留白应明显大于组件内部留白，用于建立页面节奏。
- 首屏、客户案例、价格、FAQ、表单等高价值区块应拥有更清晰的视觉分隔。
- 浅色背景区块与深色背景区块切换时，应同步校验文字对比度与按钮样式是否可读。

## 导航（Header / Nav）

- 顶部导航需优先承载品牌标识、主导航项与核心 CTA。
- 导航层级保持精简，首屏可见导航项不宜过多。
- 导航栏在桌面端应保持稳定对齐，不因文案长短产生明显跳动。
- 若使用吸顶导航，滚动前后高度变化需克制，避免突兀。
- 移动端折叠菜单应保留清晰入口与关闭反馈，不得遮挡主要内容过久。

## 首屏组件（Hero）

- 首屏必须在短时间内说明“你是谁、提供什么、为什么可信、下一步做什么”。
- 标题、说明、主 CTA、辅助 CTA 的优先级必须明确，避免首屏同时竞争多个重点。
- 首屏配图优先使用产品、品牌或业务场景真实视觉，不使用明显无关的装饰图。
- 若包含数据、客户 Logo、评分、奖项等信任元素，应靠近主信息区展示。
- 首屏按钮数量建议控制在 1 到 2 个，避免转化路径分散。

## 按钮（Button / CTA）

- 主按钮用于页面核心转化动作，如“立即咨询”“预约演示”“免费试用”。
- 次按钮用于补充动作，如“查看案例”“了解详情”“下载资料”。
- 按钮必须使用可交互语义元素，优先使用 `button` 或 `a`，不要用纯 `div` 冒充按钮。
- 主按钮使用高识别度主题色实底，次按钮使用描边或弱底色，与主按钮形成层级差。
- 按钮宽高需保证可点击性，桌面端不能做得过小，移动端需进一步放大点击区域。
- `disabled` 状态除视觉弱化外，必须真正禁用交互。
- 页面内核心 CTA 文案应保持一致，避免同一目标出现多种说法。

## 卡片（Card）

- 卡片用于承载功能亮点、服务能力、案例摘要、价格方案、团队介绍等模块化信息。
- 同一组卡片需保持统一的内边距、圆角、边框和阴影策略。
- 卡片层次建议“标题 + 描述 + 辅助信息/操作”，避免信息堆叠无主次。
- 阴影与描边二选一或弱组合，避免厚重悬浮感。
- 卡片内容长度差异较大时，应通过布局约束保持整体阵列稳定。

## 数据与信任模块（Stats / Logo Wall / Testimonial）

- 数据指标适合突出“客户数量、服务年限、转化提升、覆盖区域”等可信信息。
- 客户 Logo 墙应控制视觉整洁，尺寸、灰度或透明度策略需统一。
- 证言 / 案例卡片应优先展示结果、角色、公司身份等增强信任的信息。
- 信任模块应以建立可靠感为主；可加入少量语义化图标、评分符号、引号、认证标识等辅助元素，但不得压过核心结果信息。

## 表单（Form / Input / Textarea / Select）

- 表单服务于线索收集与咨询转化，应字段克制、路径清晰。
- 输入框默认使用浅边框或浅底色，`focus` 时以主题色高亮。
- 占位文本必须明显弱于用户输入内容，不能混淆。
- 错误状态必须同时提供边框变化与错误文案，不能只靠颜色提示。
- 必填项、提交按钮、隐私说明、成功反馈应完整可见。
- 多字段表单在桌面端可分列，但阅读与填写顺序必须自然。
- 落地页表单字段不宜过多，优先收集最关键的转化信息。

## 标签、徽标与辅助信息（Tag / Badge / Meta）

- 标签用于“热门、推荐、限时、已验证、行业方案”等短语义标记。
- 标签尺寸应小而克制，避免抢占主要内容层级。
- 成功、警告、错误、信息等语义色必须固定，不可混用。
- 辅助信息如发布时间、阅读时长、服务城市、案例行业等，建议使用低一级视觉权重。

## FAQ 与折叠项（Accordion）

- FAQ 项适合承载常见疑问、价格说明、服务边界、交付流程。
- 折叠标题需清晰可点，展开反馈明确，不得出现点击区域过小的问题。
- 默认展开策略应服务转化目标，优先展开最关键问题。
- 内容较长时要保持段落清晰，不要把 FAQ 写成整页长文。

## 页脚（Footer）

- 页脚至少应包含版权、备案、联系方式或补充导航中的核心信息。
- 页脚是补充信任和收口区域，不应承担过强的视觉竞争。
- 深色页脚需重点校验文字、链接、图标和分割线的对比度。
- 若有社媒、地址、邮箱、电话等信息，应保证样式统一、可点击。

## 图标与插画（Icon / Illustration）

- 同一页面图标风格应统一，线性与面性图标不要混用。
- 图标尺寸使用固定档位，避免局部随意缩放导致节奏失衡。
- 图标颜色优先跟随文字语义或主题色；允许少量装饰性高光、弱对比描边或浅底衬托，但应保持统一规则，避免形成无序噪音。
- 插画与配图应服务内容理解；允许使用与品牌调性一致的抽象图形、线稿、几何背景或场景化装饰，但需与业务语义保持关联。

## 状态与交互反馈

- 交互反馈应清晰但克制，优先使用透明度、阴影、位移、颜色微调等轻量方式。
- Hover 效果只增强可点击性，不改变组件核心结构。
- 弹窗、提示、成功反馈等浮层组件应优先用于补充说明，不替代页面主路径。
- 若使用模态框，必须保证可关闭、可聚焦、可回到原上下文。

## 响应式要求

- 组件在桌面端优先保证信息完整，在移动端优先保证阅读顺序与点击效率。
- 多列卡片、双栏首屏、Logo 墙、价格表在小屏下需自然折叠，不可强行压缩。
- 导航、按钮、表单、FAQ 等交互组件需在移动端放大点击区域。
- 响应式调整优先改布局与间距，不轻易删除关键信息。

## 禁止事项

- 禁止使用 APP 原型术语或设备容器表达 WEB 页面。
- 禁止大面积使用重阴影、强描边、过多渐变，导致页面厚重杂乱。
- 禁止同一页面出现多套按钮、卡片、标签、输入框风格。
- 禁止为了“丰富”而堆砌轮播、弹窗、悬浮组件，干扰主转化路径。
- 禁止使用无意义占位图、无真实语义的假数据、与业务无关或明显喧宾夺主的图标装饰。

## 快速检查清单

- 是否符合企业官网 / 落地页的 WEB 信息架构。
- 首屏是否清晰表达品牌价值、信任信息和核心 CTA。
- 导航、按钮、卡片、表单、FAQ、页脚是否风格统一。
- 是否优先使用语义化结构与真实可交互元素。
- 是否兼顾桌面端展示与移动端响应式体验。
- 颜色、间距、圆角、阴影是否复用全局变量而非散落硬编码。
- 是否避免 APP 化表达、无序视觉堆砌和转化路径分散，同时保留与品牌调性一致的适度视觉装饰。
