# 产品功能与信息结构清单（原型向）

## 何时触发

- 用户要 **功能清单**、**页面/信息结构**、**原型范围**、**MVP 分期**，或 **基于现有 PRD/需求稿整理**。
- 用户 **@ 文件** 或给出 `docs/` 等工作区内路径时，以该文档为 **真源** 先读再写。

---

## 定位

| 做 | 不做 |
|----|------|
| 把想法或需求稿整理成 **功能清单 + 页面/信息结构**，便于画原型、对齐范围与分期 | 技术方案、接口、库表、性能数值、日志/埋点、安全架构细节；默认不当开发说明书 |

若用户 **@ 文件或给路径**，必须 **摘录、归类、补全**，而非忽略原文从零重写。

---

## 目的（颗粒度）

- 按形态（营销站、Web 产品、App）分 **主题**，拆到 **用户可感知的一步**（如「邮箱登录」而非「实现 OAuth 模块」）。
- **功能 ID**（如 `WEB-010`）仅用于 **功能 ↔ 页面 ↔ 分期** 对照，不是工程接口号。
- 写清 **优先级** 与 **本期不做**，控制原型范围。

除总览表格外，须有 **结构化详细功能列表**（主题 → 子域 → 带 ID 条目），粒度以 **「原型里要有一块界面或一条流程」** 为准。

交付物 **必须** 含：**完整页面结构列表**（全站主要页/屏；规则见 **§ 完整页面结构列表**）、**树状功能清单（优先级概览）**、**关键页面布局线框图**（ASCII，至少一屏）。用户要更细时，再增加 **逐页展开与字段枚举**。

---

## 默认：简短访谈（可豁免）

**除非**用户明确说「跳过访谈 / 只要空白模板 / 信息已给全」，否则先 **5–10 个问题**，聚焦 **用户、场景、范围**，少问实现。

### 访谈提纲（按需选用）

1. 产品一句话：帮谁解决什么问题？
2. 形态：官网、需登录 Web、App（或组合）？
3. 谁在用：访客、会员、管理员等？
4. 必须先有的主路径：1～3 条用户旅程？
5. 怎样算成功：业务或体验结果（不必量化公式）？
6. 商业与合规（若有关）：付费、发票、隐私、行业限制——**只记影响范围的事实**。
7. 外部能力：登录第三方、地图、支付等——**记有无与名称**，不写对接方案。
8. 时间与边界：先看哪一版原型？**坚决不做**什么？

结论写入文件顶部 **「访谈摘要」**（要点列表）。
**豁免**：只要模板时跳过访谈，在 **「假设与待确认」** 列缺口。

---

## 用户提供需求稿时（输入式细化）

用户附上或指向需求清单 / PRD 节选 / 招标需求等时：**必须先以该文档为真源**，再与访谈合并。

### 必须执行

1. **读取**：指定文件（`.md`、`.txt` 等；多文件按用户顺序）。
2. **摘录**：功能点 / 模块 / 页面名（去重），不改原意；含糊处标 **待澄清**。
3. **映射**：每条挂靠到 **主题**、**树状功能清单** 节点、**功能总表 ID**、**完整页面结构列表** 对应屏的 **主要功能**；原文无 ID 则分配 `APP-` / `WEB-` 等并全文一致。
4. **补洞**：对照本技能必备章节，缺失部分用访谈或 **合理假设** 补齐；假设写入 **「假设与待确认」** 或访谈摘要。
5. **冲突**：输入与访谈矛盾时 **以用户最新说明为准**；无说明则在 **备注** 并列标 **待用户裁定**。

### 强烈推荐（有输入稿时）

**「需求来源对照」**（简表，便于追溯）：

| 原文件位置 | 原文要点（压缩） | 映射到（功能 ID / 页面名 / 分期） |
|------------|------------------|-----------------------------------|
| 如 `docs/xxx.md` §3 | … | `APP-010`、MVP |

原文过长时按 **章节/小标题** 聚合。无输入稿时不强制此节。

## 交付物：格式与路径

| 项 | 规则 |
|----|------|
| 格式 | Markdown（`.md`）；正式内容以文件为准，聊天可摘要 |
| 路径 | 工作区 **`docs/`**（无则创建） |
| 命名 | `docs/YYYY-MM-DD-<主题中文名>功能清单.md`，或用户指定名（仍在 **`docs/`**、`.md`） |

### 必备章节顺序（对外交付）

读者应 **先看到逐屏 IA**，再 **用树状清单对齐模块优先级**，最后进入 **ID 级功能矩阵与分期**。

1. **访谈摘要**（或 **假设与待确认**）
2. **完整页面结构列表**（必选，规则见 **§ 完整页面结构列表**）— **紧跟摘要**，为全文档 **第 2 章**；在本章 **全部逐屏小节之后**，追加 **§ 关键页面布局线框图**（必选）
3. **树状功能清单（优先级概览）**（必选，规则见 **§ 树状功能清单**）— 与 **分期**、功能总表优先级 **同一套口径**（🔴≈MVP 核心、🟡≈后续迭代、⚪≈更晚或本期不做）
4. **结构化详细功能列表**（主题 → 子域 → ID 条目）
5. **功能总表**（与详细列表 **同一套 ID**）
6. **分期**（MVP / 下一阶段 / 更晚 / 本期不做）
7. **范围与体验约束**（可选，仅当影响原型画什么时）
8. **逐页展开与字段枚举**（可选，用户要更细线框时）
9. **需求来源对照**（有用户输入稿时强烈推荐；无则省略）

起草时可先在脑中或草稿里固定 ID，但 **最终文件章节编号** 仍按上表。

**结构优化说明**：第 2 章专注「屏级 IA + 一页骨架共识」；第 3 章用树快速对齐「模块级做什么、哪一期」；第 4～6 章再落到可对照评审的 ID 与分期表，避免一上来只有表格或只有长列表。

## 工作流（执行顺序）

```text
访谈(默认) ──并行或先后──▶ 有用户稿？ ─是─▶ 读取→摘录→映射（不得未读稿覆盖用户已写功能）
                │                              │
                └───────────────┬──────────────┘
                                ▼
                         选骨架（营销站 / Web 应用 / App，见 references/taxonomy.md，输出改写成产品语言）
                                ▼
                    建议：先草 ID → 写「完整页面结构列表」(第2章) → 同章补 ASCII 关键页线框图 → 「树状功能清单」(第3章) → 详细列表 + 功能总表
                                ▼
                         分期 → 写入 docs/*.md → 自检清单
```

### 选骨架

引用 `references/taxonomy.md` 时 **改写为用户能听懂的块**，去掉纯实现项（具体协议名、统计脚本实现等），除非用户明确要求。

- **营销/内容站**：首屏、定价、留资、信任与法务、内容谁维护（不说 CMS 技术名）。
- **Web 应用**：账号、核心业务流、设置、通知/付费/后台（有则无）。
- **移动 App**：引导、主流程、推送、系统能力是否出现、上架是否本期讨论。

表壳与图例见 `references/templates.md`。

## 书写细则

### 详细功能列表（交付物第 4 章级内容）

结构：**主题 → 子域 → 列表项**，推荐 `## 主题` → `### 子域` → `- **ID** 一句话`。

每条能说清：**谁、在本产品里完成什么、大致在什么界面**。不必写字段与后端依赖。

- 粒度：用户任务 / 屏幕级能力。
- 与总表对齐：挂靠总表 `ID`；子能力可用 `APP-010-1` 后缀（可选）。
- 列表中出现的页面名，须在 **完整页面结构列表** 中有对应分屏小节；模块粒度宜与 **树状功能清单** 可对读（不必逐节点复制）。

### 功能总表（矩阵）

每行 = 一条可讨论的产品能力。可先于此草稿 ID，供页面列表引用。

| 列 | 说明 |
|----|------|
| ID | `WEB-`、`APP-`、`ADM-`、`INT-`（外部依赖仅记范围）、`NFR-`（少用） |
| 领域 | 用户理解的模块名 |
| 功能 | 短名称 |
| 用户故事 | 可选；过宽时 **优先删此整列** |
| 优先级 | MoSCoW 或 P0–P3，图例全文只写一次；**建议**与 **树状清单** 🔴🟡⚪ 互读并在文档首处说明对应关系 |
| **完成标志（原型向）** | 评审能看懂的完成描述；**不写** GWT、不写自动化脚本 |
| **与其它功能的关系** | 须先有谁、互斥、外部依赖（写 ID 或一句话）；**不写** 中间件、数据表 |
| 备注 | 分期、边界、待确认 |

按 **主题** 分组；默认 **8 列**。

### 完整页面结构列表（交付物第 2 章，必选）

**与原型/IA 对齐的主清单**：列出 **全部主要页面/屏幕**，至少覆盖 **MVP**；非 MVP 屏在 **备注** 标分期。禁止只举示例而漏主流程页（除非访谈已明确本期不做且已写入分期）。本章在全部逐屏小节之后以 **`### 关键页面布局线框图`** 收束（规则见下 **同一节内** 小节）。

#### 呈现规则（硬约束）

- **禁止**用宽表把下列五项排成多列：入口与导航、页面结构、主要功能、备注、文件名称（建议）。
- **必须**每个页面/屏幕单独一小节，标题推荐 **`### {序号}. {页面/屏幕名称（建议）}`**，其下 **无序列表**，五项键名固定（缺一不可，可写「—」占位）：

  - **入口与导航**：路由、Tab、栈、深链等 **一句**
  - **页面结构（自上而下）**：主要区块，用 **→** 或 **；** 串联，如 `顶栏 → 筛选区 → 列表 → 底栏`
  - **主要功能**：本页对应的 **功能总表 ID** + 短说明；多项 **逗号** 分隔
  - **备注**：角色可见性、跳转、线框优先级 P0/P1、待澄清
  - **文件名称（建议）**：**kebab-case**、语义短名（如 `onboarding`）；勿加 `Scr-xx-` 或与章节序号重复的标记

- **可选**：在本节开头增加 **屏目录表**（便于预览与检索）：
  - **基础**：**2 列**「序号｜页面/屏幕名称」。
  - **扩展（推荐）**：**4 列**「序号｜页面/屏幕名称｜主要功能｜文件名称（建议）」——内容须与各 `###` 小节内的 **主要功能**、**文件名称（建议）** **一致**（可适当缩写「主要功能」列以控制列宽）；表上方建议简短说明 **入口与导航、页面结构、备注** 仍以逐屏无序列表为准。
  - **硬约束**：目录表 **不得** 再增加 **入口与导航、页面结构、备注** 三列；该三项 **只能** 写在各屏的五项无序列表中。**禁止**用单张表收纳完整五项（含上述三列），等同于上条「禁止宽表五项并列」。

**完整性提示**：Web 含公共页 + 登录后核心域 + 设置/错误页；App 含启动/引导 + 主 Tab 根屏 + 常见二级栈；管理端可单独分组。

**与「逐页展开」分工**：本节负责 **全站逐屏**；逐页展开仅在少数页需要更细时追加。

#### 关键页面布局线框图（第 2 章末节，必选）

紧接在本章 **最后一个** `### {序号}. {页面/屏幕名称}` 小节之后，新增 **同级** 小节 **`### 关键页面布局线框图`**（与逐屏小节共用同一父级 `##`，例如「完整页面结构列表」；**不要**单独再开一级 `##`，以免把线框图挤出「第 2 章」）。

**目的**：在动手画高保真或开发实现之前，对 **最关键的一屏** 骨架达成共识，减少「每个人想象的布局不一致」。

**选哪一屏？** 用户 **停留时间最长** 的页，或 **核心交互发生** 的页（通常与 MVP 主路径一致）。**只画 1 屏**；若产品有 Web + App 两条主战场且差异大，可 **最多 2 屏**（各形态 1 屏），再多则移到可选「逐页展开」。

**ASCII 线框须体现：**
- 导航形态与位置（顶栏 / 左侧竖向 / 底部 Tab 等）
- 主内容区、操作区、辅助区的划分
- **视觉重心**（用简短箭头或文字标出最先被注意的区域）
- 若有：**弹窗 / 抽屉 / 侧栏覆盖层**（用文字框或备注标「覆盖层」）
- **列表 + 详情** 并存时：左右或上下的位置关系

**格式**：Box-drawing（`┌─┐│└┘`）或简易 `+---+` 均可；**标注区域名称与作用**；不要求像素级对齐。

示例（Web 后台，左侧导航）：

```
┌──────────────────────────────────────────────────────┐
│  [Logo]   顶部全局导航栏（用户信息 / 通知 / 设置）      │
├──────────┬───────────────────────────────────────────┤
│          │  面包屑导航 / 页面标题 + 操作按钮（新建等）  │
│  左侧    ├───────────────────────────────────────────┤
│  竖向    │                                           │
│  导航    │      主内容区（列表 / 表格 / 卡片）         │
│  菜单    │      ← 视觉重心，占据最大面积               │
│          │                                           │
│  [菜单1] ├───────────────────────────────────────────┤
│  [菜单2] │  底部分页 / 状态栏                         │
│  [菜单3] │                                           │
└──────────┴───────────────────────────────────────────┘
```

示例（移动端首页，底部 Tab）：

```
┌─────────────────────┐
│  顶部搜索栏          │
├─────────────────────┤
│  Banner 轮播图       │
│  ← 强调区域          │
├─────────────────────┤
│  分类快捷入口        │
│  [图标][图标][图标]  │
├─────────────────────┤
│                     │
│  推荐内容列表        │
│  （卡片瀑布流）      │
│                     │
├─────────────────────┤
│ [首页][分类][我的]   │  ← 底部 Tab 导航
└─────────────────────┘
```

**与其它章节对齐**：线框所选页面须在 **本章前文** 某一屏小节中出现；对应的 **功能总表 ID** 可在图下用一句列举（可选）。

### 树状功能清单（优先级概览，第 3 章，必选）

与「结构化详细功能列表」分工：本节是 **模块级、一棵树读完** 的执行摘要；详细列表承担 **带 ID 的可评审粒度**。

**格式**：纯文本树（`├──` `└──`），根节点为 **产品 / 项目简称**；**模块** 级节点必须带优先级前缀：

- 🔴 **核心**（MVP 必须有）
- 🟡 **重要**（后续迭代）
- ⚪ **未来规划**（更晚或本期明确不做）

叶子节点为 **用户可感知的功能名**（可与详细列表条目对应，不必写 ID；若愿意可在括号内写 `WEB-010` 等）。

**一致性**：emoji 优先级须与 **分期**、功能总表 **优先级列** 可互推；若有冲突，以 **分期与总表为准** 修正树。

示例：

```
产品名称
├── 🔴 模块A（核心，MVP 必须有）
│   ├── 功能1
│   └── 功能2
├── 🟡 模块B（重要，后续迭代）
│   └── 功能3
└── ⚪ 模块C（未来规划，暂不实现）
    └── 功能4
```

### 逐页展开与字段枚举（可选）

用户要更细线框、表单、列表列时使用。每页须重复或引用 **主要功能 ID**，并写 **关键操作**、**空态/错误**（各一句）。字段表建议列：**名称、控件类型、必填**（总列数 ≤6）。列表页可加列名与是否排序/筛选；营销页按区块写 CTA 与功能 ID。

### 范围与体验约束（可选）

默认不写长 NFR。仅当影响 **画什么、画到哪** 时简短说明，例如：是否必须手机尺寸原型；多语言/无障碍是否要考虑（不写 WCAG 等级）；合规是否影响能否收集某类信息。**不要**默认展开性能、API、安全、埋点、国际化实现。

### 分期

**MVP**、**下一阶段**、**更晚**、**本期不做** — 支持分稿与评审顺序。
